# AI 编程协作工作流（Codex 主力 + Claude/GLM5/DeepSeek + Trae + Qoder）

## 1. 目标与原则

目标：在**保证代码质量和可维护性**的前提下，最大化 AI 提效。

原则：
- 单一主控：`Codex` 负责主流程推进与最终落地。
- 多模型互证：`Claude / GLM5 / DeepSeek` 负责方案对照、风险质疑、边界补漏。
- IDE 内执行闭环：`Trae + Qoder` 负责本地改代码、跑测试、快速迭代。
- 小步快跑：每次改动都可回滚、可验证、可解释。

## 2. 工具定位（谁做什么）

- `Codex（主力）`
  - 任务拆解、实现主线、跨文件改动、提交前整合。
  - 输出可执行变更（代码+命令+验证步骤）。

- `Claude（强推理/长文本）`
  - 架构评审、复杂设计权衡、文档润色、反例推演。

- `GLM5 / DeepSeek（补充视角）`
  - 备选实现、性能优化建议、中文语境问答、快速 code review second opinion。

- `Trae + Qoder（IDE 协作层）`
  - 在本地直接应用修改、局部重构、批量替换、联动调试。
  - 与测试/构建系统形成快速反馈回路。

## 3. 标准工作流（推荐日常节奏）

### 阶段 A：任务进入（5~10 分钟）
- 输入：需求、约束、验收标准。
- 操作：
  1. 用 Codex 产出“任务卡”：背景、目标、风险、完成定义（DoD）。
  2. 复杂任务再让 Claude 产出 1 份备选方案。
- 产物：`任务卡 + 实施路线（主方案/备选）`。

### 阶段 B：实现（30~120 分钟）
- Codex 负责主实现：按模块分批改动（每批可独立验证）。
- Trae/Qoder 在 IDE 中执行：
  - 应用补丁
  - 运行单测/集成测试
  - 即时修复报错
- 每个批次结束必须记录：
  - 改了什么
  - 为什么这么改
  - 如何验证

### 阶段 C：交叉评审（10~30 分钟）
- 把关键 diff + 测试结果给 Claude 或 GLM5/DeepSeek 做二次审查：
  - 是否有逻辑漏洞？
  - 是否有边界条件遗漏？
  - 是否引入性能/安全/兼容性风险？
- 对“高风险意见”回到 Codex 做修订。

### 阶段 D：合并与交付（10~20 分钟）
- 本地执行统一门禁：lint、test、build（至少通过核心项）。
- Codex 生成最终 PR 说明：
  - 改动摘要
  - 风险点
  - 验证证据
  - 回滚方案

## 4. 三层质量门禁（必须过）

- `L1 语法与规范`：lint / 类型检查 / 格式化。
- `L2 功能正确性`：单测 + 关键路径手测。
- `L3 工程可用性`：构建通过、性能无明显回退、日志可观测。

建议：
- 小改动至少过 `L1 + L2`。
- 涉及核心链路或发布分支必须过 `L1 + L2 + L3`。

## 5. 提示词模板（直接可用）

### 5.1 给 Codex（主实现）
```text
你是本任务主开发。请按“最小可合并改动”实现：
1) 先给出变更计划（文件级）
2) 再逐步输出补丁
3) 每步附验证命令
4) 最后给风险与回滚方案
约束：不改无关文件；优先保持向后兼容。
```

### 5.2 给 Claude（架构/评审）
```text
请作为架构评审，审查以下方案与diff：
- 关注边界条件、并发一致性、可维护性、未来扩展成本
- 必须给出：高/中/低风险列表 + 修改建议 + 是否建议合并
```

### 5.3 给 GLM5 / DeepSeek（第二视角）
```text
请对这份实现做 second opinion：
- 找潜在 bug、性能瓶颈、异常处理遗漏
- 给出至少 3 条可执行优化建议
- 按“收益/改造成本”排序
```

## 6. 场景化编排（怎么选模型）

- `新功能开发`：Codex 主写 -> Claude 审架构 -> GLM5/DeepSeek 查漏。
- `疑难 bug`：DeepSeek/GLM5 先给排查树 -> Codex 实施修复 -> Claude 复盘是否彻底。
- `重构`：Claude 定边界与分层 -> Codex 分批落地 -> Trae/Qoder 快速回归。
- `性能优化`：DeepSeek/GLM5 提 profiling 假设 -> Codex 实验与落地 -> Claude 评估长期可维护性。

## 7. 分支与提交规范

- 分支命名：`feat/*`、`fix/*`、`refactor/*`。
- 提交粒度：一次提交只做一件事（可回滚）。
- 提交信息建议：
  - `feat: ...`
  - `fix: ...`
  - `refactor: ...`
  - `test: ...`
  - `docs: ...`

## 8. 风险清单（常见坑）

- 多模型意见冲突却没有“最终拍板人”（应由 Codex 主线决策）。
- 一次性让 AI 改动过大，导致不可审计。
- 只看“能跑”，忽视异常路径和回滚方案。
- 把 AI 当答案机器而不是“候选方案生成器”。

## 9. 落地建议（第一周）

- Day 1-2：只跑“小任务闭环”（< 2 小时），熟悉节奏。
- Day 3-4：引入交叉评审，形成固定 PR 模板。
- Day 5-7：统计指标（开发时长、缺陷率、回滚次数），微调分工。

建议跟踪 4 个指标：
- 需求到合并时长
- 单次改动回归失败率
- PR 一次通过率
- 线上问题回流率

## 10. 一页版执行卡（可贴团队 Wiki）

1. Codex 定计划并主实现（小步提交）
2. Trae/Qoder 本地执行与验证
3. Claude 做架构与风险审查
4. GLM5/DeepSeek 做 second opinion
5. 通过三层门禁后合并

如果只能保留最小闭环：
`Codex 实现 -> IDE 验证 -> 另一模型复审 -> 合并`
