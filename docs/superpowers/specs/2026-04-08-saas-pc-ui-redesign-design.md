# SaaS PC端UI重设计方案（`ui/` 目录）

## 1. 背景与目标
当前项目在 `ui/` 下已有一套静态高保真原型（`index.html + iframe + pages/*`），但存在以下问题：
1. 视觉语言一致性不足，页面之间设计节奏不统一。
2. 文案与编码表现存在异常（部分中文显示乱码），影响评审可读性。
3. 组件样式复用度不够，后续映射到前端工程时成本偏高。

本次重设计目标：
1. 在不改变 `ui/` 原型信息结构基础上，重建一套统一、可扩展的 PC 端 SaaS 视觉系统。
2. 风格定位为“轻科技版”，兼顾企业感与现代感。
3. 采用“标准信息密度”：4个核心指标 + 2个趋势区 + 2个列表区。
4. 覆盖全量页面：`index + dashboard + tenant + rbac + billing + apikey + usage + ai-gateway + login + register`。

## 2. 需求确认结论
本次与用户确认后的固定约束如下：
1. 改动范围仅限 `D:\develop\CodeProject\jay-boot\ui`，不改 `jay-boot-frontend`。
2. 风格方向选择：`轻科技版`。
3. 首页密度选择：`标准密度`。
4. 页面范围选择：`全量页面`。

## 3. 方案对比与决策
### 3.1 备选方案
1. 方案A 轻科技卡片化（推荐）
2. 方案B 高对比数据台
3. 方案C 品牌展示型控制台

### 3.2 选择理由
最终选择方案A，原因如下：
1. 与“轻科技”目标一致，兼具可读性与企业后台稳重感。
2. 支持后续模块化扩展，适合作为 SaaS 脚手架长期演进基线。
3. 在视觉提升与实施复杂度之间平衡最好，能快速形成统一规范。

## 4. 信息架构设计
### 4.1 总体骨架
采用三段式 PC 控制台骨架：
1. 左侧导航：模块入口与分组。
2. 顶部状态栏：面包屑、环境状态、租户上下文、用户头像。
3. 主内容区：承载 iframe 页面，统一页面模板与视觉节奏。

### 4.2 导航分组
1. 总览：工作台。
2. 租户与权限：租户管理、角色权限。
3. 商业化：订阅计费、用量统计。
4. 开发者：API Key、AI 网关。
5. 认证：登录页、注册页。

### 4.3 页面模板（业务页统一）
每个核心业务页统一为四段：
1. `Page Header`：标题、说明、主次操作。
2. `KPI Row`：4个关键指标卡。
3. `Analysis Row`：2个趋势或分析面板。
4. `List Row`：2个表格或列表面板。

## 5. 视觉系统（Design Tokens）
### 5.1 字体系统
1. 英文与数字：`Plus Jakarta Sans`。
2. 中文：`Noto Sans SC`。
3. 回退字体：`PingFang SC`、`Microsoft YaHei`、`sans-serif`。

### 5.2 色彩系统
1. 主品牌色：`#0A84FF`。
2. 品牌强调色：`#0B5FCC`。
3. 成功色：`#16A34A`。
4. 警告色：`#D97706`。
5. 危险色：`#DC2626`。
6. 主文本：`#0F172A`。
7. 次文本：`#475569`。
8. 页面底色：`#F3F7FB`。
9. 卡片底色：`#FFFFFF`。
10. 边框：`#DCE6F0`。
11. 弱分割线：`#EAF0F6`。

### 5.3 空间与形态
1. 基础栅格单位：4px。
2. 常用间距：8 / 12 / 16 / 20 / 24。
3. 圆角：10（控件）/ 14（卡片）/ 18（容器）。
4. 阴影：
   - Card：`0 8px 24px rgba(15,23,42,0.06)`
   - Pop：`0 14px 36px rgba(15,23,42,0.10)`

### 5.4 背景策略
采用双层渐变 + 轻网格纹理，提供科技感但不干扰正文阅读，不使用高饱和大面积炫光。

## 6. 组件规范
### 6.1 通用组件族
1. `PageHeader`：统一标题区，支持主次按钮。
2. `KpiCard`：指标名、主值、趋势（up/down/flat）。
3. `PanelCard`：承载图表、表格、表单等内容。
4. `StatusBadge`：`success / warn / error / neutral / info`。
5. `DataTable`：统一表头高度、行分隔、hover 反馈。
6. `ActionBar`：搜索、筛选、导出、创建等操作入口。

### 6.2 交互规则
1. hover 以颜色、边框、阴影变化为主，不做明显位移抖动。
2. 点击组件统一 `cursor: pointer`。
3. 焦点态统一蓝色 focus ring，保证键盘可访问性。
4. 动效时长 160ms 至 220ms，避免超过 300ms。

### 6.3 层级规则
z-index 采用有限刻度：`10 / 20 / 40 / 60`，不使用随意大值，避免层叠冲突。

## 7. 全量页面落地说明
### 7.1 `index.html`（壳层）
1. 重构侧栏分组与选中态。
2. 顶栏胶囊状态标签视觉统一。
3. iframe 容器边框、圆角、阴影统一。

### 7.2 `dashboard.html`
1. 4 KPI 核心指标卡。
2. 2 个趋势分析区。
3. 2 个列表区（待办与告警）。

### 7.3 `tenant.html`
1. 租户概览卡。
2. 租户列表表格与状态标签。
3. 操作区保持与全局按钮体系一致。

### 7.4 `rbac.html`
1. 角色列表区。
2. 权限矩阵区。
3. 变更记录区。

### 7.5 `billing.html`
1. 套餐摘要卡。
2. 账单趋势区。
3. 流水与状态表格区。

### 7.6 `apikey.html`
1. Key 列表区。
2. 创建 Key 的表单/弹层样式区。
3. 权限说明与风险提示区。

### 7.7 `usage.html`
1. 用量 KPI 与配额进度。
2. 调用趋势区。
3. 调用明细表。

### 7.8 `ai-gateway.html`
1. 模型路由状态区。
2. 延迟/成本趋势区。
3. 限流或告警区。

### 7.9 `auth-login.html`
1. 品牌侧信息区 + 登录表单卡。
2. 与主控台共享颜色、字体与圆角体系。

### 7.10 `auth-register.html`
1. 注册表单与协议提示。
2. 与登录页保持视觉同源。

## 8. 状态与异常展示规范
1. 每个关键数据模块应具备：加载态、空态、错误态占位。
2. 错误信息使用统一 `danger` 提示容器，不使用散乱红字。
3. 空态文案简洁，提供下一步建议动作（如“刷新”“返回上一步”）。

## 9. 响应式与兼容策略
1. 重点优化断点：1440 / 1200 / 1024。
2. 768 以下自动降级为单列布局。
3. 保证无横向滚动条（特殊表格区域除外，需容器内横向滚动）。
4. 支持 `prefers-reduced-motion`，减少动画干扰。

## 10. 实施边界与约束
1. 本设计文档仅针对 `ui/` 静态原型目录。
2. 不调整项目结构，不删除现有文件。
3. 不修改 `package.json`、`Dockerfile`、部署脚本。
4. 不引入 TODO、伪代码，保证可直接打开预览。

## 11. 验收标准
1. `ui/index.html` 可直接打开并稳定切换所有子页面。
2. 10 个页面视觉语言一致，信息层级清晰。
3. 主交互具备 hover、focus、active 基础反馈。
4. 中文文案显示正常，不出现乱码。
5. 页面在 1440、1200、1024、768 断点均可用。

## 12. 风险与应对
1. 风险：历史文件存在编码问题导致中文再次异常。
   - 应对：统一以 UTF-8 无 BOM 写入并复检关键文案。
2. 风险：页面样式重复定义导致视觉漂移。
   - 应对：强化 `global.css` 的 token 与通用组件类复用。
3. 风险：iframe 子页独立样式差异过大。
   - 应对：将结构类与状态类收敛到全局样式，页面仅做少量差异扩展。

## 13. 后续实施建议
1. 先改 `index.html + assets/global.css` 建立全局框架。
2. 再按页面优先级批量改造：`dashboard -> tenant/rbac -> billing/apikey/usage/ai -> auth`。
3. 完成静态稿后，再映射到 `jay-boot-frontend` 真实 Vue 页面。
