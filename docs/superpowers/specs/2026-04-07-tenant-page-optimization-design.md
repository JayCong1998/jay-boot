# 租户管理页优化设计（2026-04-07）

## 1. 背景与目标

当前前端租户管理页可完成“查看当前租户信息 + 修改租户名称”，但信息呈现偏技术化，业务可读性不足。  
本次优化目标是：在不新增后端接口的前提下，提升“租户核心信息首屏可读性”和“名称修改交互体验”。

## 2. 范围与约束

### 2.1 本次范围（In Scope）

- 页面：`jay-boot-ui/src/views/TenantView.vue`
- 接口：
  - `GET /api/tenants/current`
  - `POST /api/tenants/current/update`
- 视觉目标：业务化表达优先（少字段、中文标签、重点信息突出）
- 交互目标：加载态、错误态、提交态可感知

### 2.2 非范围（Out of Scope）

- 不新增成员管理、安全策略、时区等配置能力
- 不新增后端接口
- 不调整全局布局与路由结构

## 3. 方案选择结论

已评估 3 个方向：

- A：业务概览 + 基础设置（推荐并已确认）
- B：单页详情（信息面板化）
- C：左右分栏（左信息右操作）

最终采用 A，原因：

- 与“信息呈现优先 + 业务化风格”目标一致
- 完全适配现有后端能力
- 改造成本低、上线风险小

## 4. 页面信息架构

页面拆分为 3 个层级区域：

1. 顶部区
- 标题：租户管理
- 说明：维护当前工作区核心配置
- 操作：刷新按钮（重新拉取租户信息）

2. 业务概览区（3 张卡片）
- 卡片 1：工作区名称（主视觉）
- 卡片 2：订阅套餐（`planCode`）
- 卡片 3：负责人（以业务文案展示，保留 `ownerUserId`）

3. 基础设置区
- 仅保留“工作区名称修改”一项
- 提交按钮放在表单操作区，避免与概览信息混杂

## 5. 组件与状态设计

## 5.1 状态定义

- `loading`: 页面首次加载态
- `saving`: 提交名称修改中
- `loadError`: 页面内错误信息（用于显示错误态和重试）
- `tenant`: 当前租户信息对象（`tenantId/name/ownerUserId/planCode`）
- `nameInput`: 名称编辑值

## 5.2 关键交互

1. 初始进入
- 触发 `currentTenantApi()`
- 成功：填充概览与输入框初始值
- 失败：显示页面内错误态 + 重试按钮

2. 手动刷新
- 触发 `currentTenantApi()`
- 刷新时不强制覆盖“已修改但未提交”的输入（仅在无改动时同步）

3. 修改提交
- 提交前校验：
  - `trim` 后非空
  - 长度 `2-64`
- 成功：同步租户信息、提示成功
- 失败：保留输入值，提示失败原因

4. 防重复提交
- `saving=true` 时禁用提交按钮并显示 loading

## 6. 数据流设计

1. 页面挂载 -> `loadTenant(false)`  
2. `loadTenant` 调用 `currentTenantApi()` -> 返回 `TenantCurrentResponse`  
3. 数据映射到 `tenant`，并按同步策略处理 `nameInput`  
4. 用户提交 -> `updateCurrentTenantApi({ name })`  
5. 返回后更新 `tenant` 与输入值、提示结果

## 7. 错误处理策略

- 请求失败统一使用 `message.error` 反馈
- 首屏加载失败采用“页面内错误态”而非仅 toast，保证可恢复路径清晰
- 提交失败不清空输入，减少二次输入成本

## 8. 视觉与文案规范

- 文案优先业务术语：工作区名称、订阅套餐、负责人
- 技术字段弱化展示：`tenantId/ownerUserId` 放在次级信息
- 空值统一显示 `-`
- 复用现有 `global.css` 视觉变量，不引入新主题体系

## 9. 响应式要求

- 桌面端：三卡片并排
- 窄屏：卡片自动单列，表单纵向排列
- 输入与按钮在移动端保持足够点击区域

## 10. 验收标准

- 可稳定展示当前租户 4 个核心字段
- 名称校验与后端约束一致（必填、2-64）
- 首屏加载失败可在页面内重试恢复
- 提交过程具备 loading 和防重复点击
- 页面在桌面与移动端均可正常浏览与操作

2. 增加加载态、错误态、刷新与重试逻辑
3. 增加前端校验规则并完善提交交互反馈
4. 本地构建验证 `npm run build`
